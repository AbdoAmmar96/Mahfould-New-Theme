### 1.1 تسجيل المزوّدين والموافقة (حسم سؤال 1) ✅
- **صفحة تسجيل مستقلة** للمزوّدين، **مختلفة تمامًا** عن تسجيل العملاء العاديين.
- عند التسجيل **يُنشأ الحساب تلقائيًا فورًا** → المزوّد يقدر يدخل لوحته (`/vendor`) ويجهّز خدماته.
- **لكن الخدمات لا تُنشر على الموقع للعملاء إلا بعد موافقة الـSuper Admin.**
- حالة المزوّد: `pending` (بعد التسجيل) → `approved` (نشر مسموح) / `rejected` / `suspended`.
- حالة كل خدمة/إعلان: `draft` → `pending_review` → `published` / `rejected`.
- **الأفراد:** رفع **"فيش وتشبيه"** (شهادة السجل الجنائي) ضمن مستندات التوثيق قبل الموافقة.

## 2) كتالوج الخدمات (7 أنواع)

الرحلات · الفنادق · المطاعم · العربيات · الباصات · التوصيل · **التخصيص (صاحب السعادة)**.

كل خدمة تنتمي لـ **مزوّد** (طرف أول = محفول مكفول، أو طرف ثالث = شركة/فرد)، ولها **تقييم على مستوى العنصر** + مساهمة في **تقييم المزوّد المجمّع**.

---

## 3) نموذج المزوّد (طرف أول / طرف ثالث)

- **طرف أول (محفول مكفول):** خدمات تحمل **ضمان "مكفول"**، لها الأولوية والشارة.
- **طرف ثالث:** شركات + أفراد، بحساب + flow + **بروفايل عام** (لوجو + كل التقييمات على كل عملياته المجمّعة).
- **علم الضمان مستقل عن التوثيق:**
  - `is_verified` = **موثّق** (مستندات/ترخيص سليم).
  - `is_first_party` / `guaranteed` = **مكفول** (ضمان محفول مكفول) → هو ما يمنح الشارة وأولوية البحث.
- **تقييم المزوّد المجمّع:** `provider_review_score` + `provider_review_count` يُحسبان من كل عناصره.

## 4) نظام الحجز الأساسي (موحّد لكل الخدمات)

1. **تاريخ + عدد الأفراد**.
2. لو العدد أكبر من الخيارات السريعة → **زر "المزيد"** يفتح **popup**: العدد + **عمر كل فرد**.
3. **تسعير حسب شرائح العمر** (رضيع / طفل / بالغ) — جدول شرائح لكل خدمة.
4. **الحجز لنفسه أو لطرف آخر.**
5. **بيانات المستفيد** (حسم سؤال 5) ✅: **الاسم + الرقم القومي + السن — للشخص الرئيسي فقط** (أما الأعمار فتُجمَّع لكل الأفراد لأغراض التسعير).
6. **قاعدة أمان:** لو الحجز **لطرف آخر** → **الدفع أونلاين كامل إلزامي** (لا "عند الاستخدام" ولا مقدّم فقط).
7. **تقويم اختيار تاريخ محسّن** (تجربة أفضل) عبر مكوّن `DatePicker`/`DateRangePicker` من Shadcn.

## 5) مبدأ الدفع الموحّد (حسم سؤال 2) ✅

**الأصل: الدفع عند نقطة الخدمة** (الوصول للفندق · استخدام الباص/العربية · المطعم). **الدفع الأونلاين المسبق اختياري** للعميل — و**إلزامي فقط لو الحجز لطرف آخر** (استثناء أمني). **لا مقدّم في أي مسار — عدا الرحلة المخصّصة (صاحب السعادة) التي تتطلب مقدّمًا عند الحجز.**

| الخدمة | الأصل (لنفسه) | ملاحظة |
|---|---|---|
| **الفندق** | **لا دفع إلا عند الوصول للفندق — مبدأ أساسي (بلا مقدّم إطلاقًا)** | مسبقًا فقط لو لطرف آخر |
| **الرحلة** | مصاريف الخدمة **لا تُدفع إلا عند الوصول للفندق** | كل مكوّن (باص/عربية/مطعم) بنفس مبدئه |
| **الباص** | يُدفع **عند الباص** (وقت الركوب) | إلا لو دفعها مسبقًا |
| **العربية** | تُدفع **عند الاستخدام** | إلا لو دفعها مسبقًا |
| **المطعم** | يُدفع **عند المطعم** | إلا لو دفعها مسبقًا |
| **التوصيل** | يُدفع **عند الاستخدام** (بالكيلومتر) | إلا لو دفعها مسبقًا |
| **الرحلة المخصّصة (صاحب السعادة)** | **مقدّم عند الحجز** | مركّبة من عدة مزوّدين (حسم سؤال 3) ✅ |
| **أي حجز لطرف آخر** | **كامل أونلاين مسبقًا — إلزامي** | يتجاوز كل ما سبق |

> إمّا **دفع كامل عند نقطة الخدمة** (الأصل)، أو **دفع كامل مسبق** (اختياري للعميل / إلزامي لو لطرف آخر). الفندق **دائمًا** عند الوصول (ما لم يكن لطرف آخر — وقتها كامل مسبقًا).

## 6) تدفّق النقل داخل حجز الرحلة/الفندق (حسم سؤال 7) ✅

عند حجز **رحلة أو فندق**، بعد اختيار الخدمة تظهر **3 خيارات نقل**:

```
        ┌─────────────────────────────────────────────┐
        │   إزاي هتوصل؟ (اختر واحد)                    │
        ├─────────────────────────────────────────────┤
        │ 1) هوصّل بعربيتي     → يتولّد QR code للدخول │
        │ 2) أحجز باص          → صفحة الباصات/الخطوط   │
        │ 3) أحجز عربية        → geolocation → أقرب    │
        │                        مدينة/نقطة استلام     │
        └─────────────────────────────────────────────┘
```

- **الخيار 1 (بعربيته):** يُصدر **QR code** عند حجز الرحلة، **يُرسَل للعميل ويظهر دائمًا في بروفايله** (حسم سؤال 6) ✅، ويُستخدم كـ**تصريح دخول** عند الوجهة.
- **الخيار 2 (باص):** يدخل مسار حجز الباص (§10).
- **الخيار 3 (عربية):** **geolocation** → **أقرب مدينة/نقطة استلام** (haversine على `locations.lat/lng` بلا API خارجي) أو مدينة يختارها → عرض العربيات المتاحة من **أسطول المنصة المملوك + المزوّدين** (حسم سؤال 4) ✅ بلوجو/بروفايل.

## 7) الفنادق

- تجربة **أشبه بـ Booking**: بحث (وجهة + تواريخ + أفراد) → قائمة بفلاتر → صفحة الفندق → اختيار غرفة → دفع.
- **بنية غرف:** الفندق كيان أب، وله **أنواع غرف (`room_types`)** يحمل كل نوع الكمية والسعر/الليلة وسياسة الإلغاء والإفطار… (الشاليه = نوع غرفة بكمية 1).
  - *ملاحظة تنفيذ:* الجدول الحالي مسطّح (سعر على صف الفندق) → يلزم **جداول جديدة + ترحيل** السعر الحالي لغرفة افتراضية واحدة لكل فندق قائم.
- **الدفع: لا شيء حتى الوصول للفندق — مبدأ أساسي** (§5). بلا مقدّم (إلا لو الحجز لطرف آخر → كامل مسبقًا).
- **القناة:** يدخلها **الأدمن + مزوّدون** (حسم سؤال 5) ✅.
- **سياسة الإلغاء:** تُعرض بالتواريخ الفعلية، و**تُحفظ snapshot وقت الحجز** (لا تتأثر بتعديل المنشأة لاحقًا).
- **no-show (حسم سؤال 7):** يُصادَر **كامل** المبلغ المدفوع مسبقًا (إن وُجد). وفي مسار "الدفع عند الوصول" (بلا دفع مسبق) لا مبلغ يُصادَر → يُسجَّل كأثر على **موثوقية العميل** (وقد يُطلب دفع مسبق من متكرري التخلّف).

## 8) الرحلات

- **طرف أول (محفول مكفول)** بضمان مكفول · **+ شركات طرف ثالث** بحساب/بروفايل تعرض رحلاتها.
- **صفحة الرحلة:** الاسم + التقييم + **تعليقات المستخدمين**.
- **بروفايل الشركة:** كل التعليقات على **كل عملياتها** (مجمّع).
- **التخصيص عبر "صاحب السعادة" (حسم سؤال 3) ✅:** **بنّاء ذاتي فوري** — العميل يركّب رحلته بأسعار جاهزة لحظيًا (لا طلب عرض سعر). يوسّع نموذج `SahbPackage` الموجود. **الدفع: مقدّم عند الحجز** (استثناء عن مبدأ "عند الوصول" لأنها مركّبة من عدة مزوّدين).
- **إثراء كل رحلة:** **فعاليات اختيارية** (add-ons بسعر) + **مخطط زمني** (itinerary يوم بيوم).

## 9) المطاعم

- **بحث واعٍ بالموقع:** أقرب لك (geolocation أو آخر موقع بالبروفايل) · الأعلى تقييمًا · **بحث بالاسم** · خيارات أخرى (مطبخ/نطاق سعر).
- **الافتراضي:** الأقرب لموقعك، أو **مطاعم زرتها وأعطيتها تقييمًا عاليًا** (تخصيص §12).
- **اختيار الترابيزة** فعليًا (العميل يعرف/يختار ترابيزته).
- **كاتلوج/منيو** خاص بكل مطعم.
- **تمييز مطعم / كافيه** (نوع المكان).
- **الحجز بفترات زمنية** بسعة (slot capacity).
- **إيضاح الرسوم/الضريبة:** هل السعر **شامل** أم **غير شامل** رسوم الخدمة/الضريبة — يظهر صراحة.
- **الدفع:** عند المطعم وقت الاستخدام، إلا لو دُفع مسبقًا (§5).
- **القناة:** يدخلها **الأدمن + مزوّدون** (حسم سؤال 5) ✅.

## 10) الباصات

- **صفحة باصات** + **صفحة تجميعية** تعرض **كل الرحلات/الخطوط مجمّعة** للعميل.
- **شركات بلوجو** بجانب كل خدمة.
- **خطوط سير (`bus_routes`)** يدخلها **الأدمن أو المزوّد**: **موعد بدء/انتهاء** + **محطات** مرتّبة.
- **التسعير بالمناطق (Zones) — حسم سؤال 2** ✅: الأجرة حسب المناطق التي يعبرها الراكب بين محطتي الركوب والنزول (نموذج مناطق، لا مصفوفة كاملة محطة→محطة).
- **مقاعد/سعة + مواعيد إقلاع ثابتة — حسم سؤال 3** ✅: كل خط له **رحلات مجدولة** بمواعيد ثابتة و**سعة مقاعد**؛ العميل **يحجز مقعدًا** على رحلة محددة.
- **الدفع:** عند الباص وقت الركوب، إلا لو دُفع مسبقًا (§5).

## 12) البحث والتخصيص

- **"أفضل قيمة مقابل السعر" (حسم سؤال 4) ✅ — داخل النوع الواحد فقط** (رحلة×رحلة، فندق×فندق…). **لا مقارنة عبر الأنواع** (تتجنّب تطبيع الأسعار).
  - `ValueScoreService`: **درجة القيمة = (وزن الفعاليات + وزن الضمان) ÷ السعر** → ترتيب "الأنسب" + شارة "أفضل قيمة" (بدل "أفضل سعر" = الأرخص).
- **طبقة التخصيص (Personalization):** موقع المستخدم (لحظي/محفوظ) + تاريخ زياراته/حجوزاته + **تقييماته هو** → تغذّي الترتيب الافتراضي (أوضح مثال: المطاعم).
- **البروفايل** يحفظ **عناوين متعددة** (البيت/الشغل/الساحل…) يختار منها العميل ✅.

## 14) محرك الإتاحة الصحيح (العمود الفقري التقني)

> مستخلص من مراجعة عدائية (39 وكيلًا) صحّحت عيوب النسخة السابقة. القاعدة: **مستحيل overbooking فعلًا**، لا شعارًا.

- **المخزون دفتر لا عدّاد:** المتبقّي = عدد الصفوف النشطة، لا عمود `booked` مخزّن.
- **صف لكل وحدة فيزيائية لكل ليلة (`unit_index`)** — **لا** عمود `qty` عدّاد. ده اللي يخلّي الـ**unique index يشتغل فعليًا** لمخزون الكمية (فندق فيه N غرفة).
- **الـUNIQUE index الحاسم:** عمود مولّد `active_flag` (= 1 نشط، NULL بعد التحرير) على `(unit_type, unit_id, date, slot, unit_index, active_flag)` → النشط مستحيل يتكرّر، والمحرّر يختفي من القيد. (يعمل على MySQL 8 و SQLite ≥ 3.31 — **ممنوع** partial index لأن MySQL لا يدعمه.)
- **قفل على مستوى الوحدة لا المدى:** `hold:unit:{id}` يغطّي كل الليالي (يمنع ثغرة المديات المتداخلة في ليلة مشتركة).
- **3 طبقات حقيقية:** قفل ذرّي (redis/database) → transaction + إعادة تحقّق لكل ليلة → **unique index** (خط الدفاع الأخير الموجود فعلًا).
- **`convert()` يعيد التحقّق داخل القفل + الترانزاكشن** قبل التأكيد؛ ولو ليلة لم تعد متاحة → `pending_review` + إخطار أدمن + بدء استرداد (لا يُحيي قفلًا محرَّرًا فوق حجز آخر).
- **حالة `pending_review`** ضمن enum + آلة الحالة + الإشعارات.
- **`ReleaseExpiredHolds`** كل دقيقة (مراقبة إلزامية — لو وقف يتقفل المخزون)، مع علامة `payment_in_progress` تمنع إلغاء hold وسط دفعة.
- **الأداء:** `where('date',…)` لا `whereDate()` (حفاظًا على الفهرس)؛ واختبار التزامن على **MySQL** لا SQLite.
- **الدقة المالية:** `decimal(12,2)`؛ **snapshot للسعر وسياسة الإلغاء** وقت الحجز؛ العمولة تُفصل عن رسوم الخدمة (لا ازدواج محاسبي).

## 15) لوحات التحكم (React عبر CrudController — لا Filament)

- **`/admin` (Super Admin):** كل الحجوزات · **موافقة/رفض المزوّدين وخدماتهم** · فحص المستندات (فيش وتشبيه) · إدخال خطوط الباصات + تسعير المحطات · ضبط الرسوم/الضرائب/أوزان "أفضل قيمة" · **إضافة موظفي الدعم** · التسويات · التقارير · **`/admin/analytics` — لوحة إحصائيات ورسوم بيانية** (تفاصيل §15.1).
- **`/support` (دعم فني):** تذاكر/محادثات العملاء · متابعة حجز/استرداد/استفسار · تصعيد — بلا صلاحيات مالية/موافقات.
- **`/vendor` (المزوّد):** تجهيز خدماته (تظل `pending` حتى الموافقة) · التقويم والإتاحة · حجوزاته · **مسح QR الدخول** للعملاء الجايين بعربياتهم (بكاميرا الموبايل من اللوحة) · الردّ على التقييمات · أرباحه وتسوياته.

### 15.1 لوحة إحصائيات الأدمن — `/admin/analytics`

**الأداة الرسومية:** **Chart.js** (خفيفة، Canvas، أسرع للأرقام والمخططات القياسية) — و**D3.js** فقط للمخططات المخصّصة (خرائط حرارية، Sankey لتدفّق العملاء، شبكات) عندما لا تكفي Chart.js.

**المؤشرات الأساسية (KPI Cards — 4 كروت في الأعلى):**
- إجمالي العملاء المسجّلين (مع Δ الأسبوع/الشهر)
- إجمالي الحجوزات (بحسب الحالة)
- الإيرادات (شامل + عمولة محفول)
- تقييم متوسّط للمنصّة

**الرسوم البيانية (Chart.js):**
- **Line chart** — الحجوزات والإيرادات يومياً/أسبوعياً/شهرياً (اختيار مدى زمني).
- **Bar chart** — الحجوزات لكل نوع خدمة (رحلات/فنادق/مطاعم/عربيات/باصات/توصيل/سعادة).
- **Doughnut chart** — توزيع طرق الدفع (كارت/محفظة/عند الوصول).
- **Bar chart أفقي** — أعلى 10 مزوّدين بالإيرادات.
- **Line chart** — نموّ العملاء الجدد شهرياً.
- **Bar chart** — تذاكر الدعم بحسب النوع/الأولوية.

**الرسوم البيانية (D3.js — للحالات المتقدّمة):**
- **Heatmap** — كثافة الحجوزات حسب اليوم × الساعة (لفهم أوقات الذروة).
- **Choropleth بسيط** لخريطة مصر بتوزيع الحجوزات حسب المحافظة (لو `locations` فيها بيانات محافظة).

**فلاتر الصفحة:** المدى الزمني (اليوم/الأسبوع/الشهر/العام/مخصّص) · نوع الخدمة · المزوّد · المدينة.

**API-first:** كل الرسوم بتقرأ من `AdminAnalyticsController` endpoints ترجع JSON — تسهيل التصدير كـCSV/Excel وتجديد الرسوم بلا refresh الصفحة.

**تفاصيل التنفيذ:** `chart.js` + `react-chartjs-2` عبر npm (لا CDN — كل شيء محلي بلا اعتماد خارجي). `d3` عبر npm بس للـHeatmap والخريطة. توكنات الألوان من `tailwind.config.js` (كورال/نيفي/مكفول/VIP) لتناسق مع الهوية.

## 17) User Flows لكل مستخدم

**زائر:** تصفّح → بحث/مقارنة (قيمة/موقع) → عند الحجز يُطلب تسجيل → يكمل كعميل.

**عميل:** تسجيل → بحث مخصّص (موقعه + تاريخه + تقييماته) → صفحة الخدمة (تقييمات + بروفايل المزوّد) → تاريخ/عدد/أعمار (popup) → **لنفسه أو لطرف آخر** → *(رحلة/فندق: اختيار النقل — بعربيته=QR / باص / عربية=geolocation)* → **دفع حسب مصفوفة §5** → تأكيد + كود + QR → بعد الخدمة **يقيّم**.

**Super Admin:** لوحة → إدارة الخدمات/الحجوزات → **مراجعة المزوّدين** (فحص فيش وتشبيه/مستندات → موافقة تُتيح النشر) → إدخال خطوط الباصات + تسعير المحطات → ضبط الرسوم/الأوزان → **إضافة موظفي دعم** → التسويات.

**دعم فني:** دخول (حساب أنشأه الأدمن) → تذاكر/محادثات → متابعة/تصعيد — بلا صلاحيات مالية.

**مزوّد (شركة/فرد):** **تسجيل من صفحة المزوّدين المنفصلة → الحساب يُنشأ فورًا** → رفع المستندات (الأفراد: فيش وتشبيه) → تجهيز الخدمات (`draft`) → **إرسال للمراجعة** → **موافقة الأدمن تنشرها** → استقبال حجوزات → تسويات + ردّ على التقييمات.

## 18) القرارات المحسومة (إجابات الأسئلة) ✅

| القرار | الحسم النهائي |
|---|---|
| تسجيل المزوّد | صفحة منفصلة · **حساب تلقائي فورًا** · **النشر بعد موافقة الـSuper Admin** |
| **دفع الفندق/الرحلة** | **لا دفع إلا عند الوصول للفندق — أساسي (بلا مقدّم)** |
| دفع الباص/العربية/المطعم/التوصيل | **عند الاستخدام**، إلا لو دُفع مسبقًا (اختياري) |
| الحجز لطرف آخر | **كامل أونلاين مسبقًا — إلزامي** |
| صاحب السعادة | **بنّاء ذاتي فوري** بأسعار جاهزة · **دفع مقدّم عند الحجز** |
| "أفضل قيمة" | **داخل النوع الواحد** فقط |
| بيانات المستفيد | اسم + رقم قومي + سن **للرئيسي فقط** (الأعمار للكل للتسعير) |
| تحويل Shadcn | **كل الصفحات** |
| النقل عند حجز رحلة/فندق | 3 خيارات (بعربيته=QR / باص / عربية) |
| **الـQR** | يُصدر عند حجز الرحلة · يُرسَل للعميل · يظهر في بروفايله |
| **تسعير الباص** | **بالمناطق (Zones)** |
| **حجز الباص** | **مقاعد/سعة + مواعيد إقلاع ثابتة** (يحجز مقعدًا) |
| **العربيات** | **أسطول مملوك + مزوّدون** |
| **الفنادق/المطاعم** | **يدخلها الأدمن + مزوّدون** |
| **no-show الفندق** | يُصادَر **كامل** المدفوع مسبقًا (إن وُجد) |
| **البروفايل** | **عناوين متعددة** يختار منها العميل |
| **قارئ الـQR** | **المنشأة تمسحه من لوحتها** (كاميرا الموبايل) |
| **الرحلة المخصّصة** | **مقدّم عند الحجز** |

## 19) استنتاجات من المراجعة العدائية (ما نتجنّبه)

- **لا** Filament / Redis-كإلزام / MariaDB في التوثيق — الستاك الفعلي فقط.
- **لا** إعادة إنشاء `bookings` — ALTER فقط، و**إنشاء `companies` أولًا**.
- **لا** عمود `qty` عدّاد — بل `unit_index` (وإلا الـunique index يكسر مخزون الفنادق).
- **لا** قفل بمدى التواريخ — قفل على مستوى الوحدة.
- **لا** `convert()` بلا إعادة تحقّق داخل قفل.
- **لا** تعليم مراحل "تمّت" بلا كود يقابلها.
- **لا** ازدواج محاسبي (عمولة على رسوم الخدمة) · **لا** سياسة إلغاء بلا snapshot · **لا** `no_show`/تأمين مسترد بلا حالة تتبّع.
- **لا** ثلاثة معانٍ لكلمة "مكفول" — فصلنا التوثيق عن الضمان.
- **لا** مسارات حجز مفتوحة بلا حماية — rate-limit + حد holds.
